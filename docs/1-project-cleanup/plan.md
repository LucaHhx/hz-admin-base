# project-cleanup

> 将项目从业务系统清理为可复用的通用后台管理基础模板（HAB）

## 目标

1. 移除所有业务特定代码和示例代码，使项目成为一个干净的后台管理基础底座
2. 统一项目标识为 HAB，便于作为新项目的起点快速复用
3. 保留并规范化业务扩展入口（business / api），支持后台管理和客户端 API 两种业务模式

## 范围

**包含:**

- 移除 example 模块（客户管理、文件上传下载示例、断点续传示例）的前后端代码
- 移除插件模块（announcement 公告、email 邮件）及其初始化注册代码
- 清理前端相关残留：移除与 example、插件模块对应的前端页面（view 目录）、前端路由配置、前端 API 调用文件、前端组件等
- 清理 business 目录残留（如已删除的 b_batch_pay.go），保留规范的空壳入口结构
- 统一项目标识：Go 模块名、前端 package name、配置文件中的项目名称等全部改为 HAB 相关标识
- 清理 .agents/skills/ 下已废弃的旧 skill 文件（code-generator、mysql-operator、skill-creator）
- 更新 README 为通用模板说明
- 更新配置示例文件（config.example.yaml）中的项目名称
- 移除 Redis 强制依赖：模板项目默认使用 SQLite，Redis 作为可选组件而非必须

**不包含:**

- 系统核心模块（用户管理、角色权限、菜单管理、API 管理、字典、代码生成器等）不做任何修改
- 不涉及功能新增或架构调整
- 不涉及前端框架升级或技术栈变更

## 核心用户场景

- 场景 A: 开发者 fork/clone 此项目作为新业务系统的起点，项目中没有任何旧业务残留代码，开箱即用
- 场景 B: 开发者在 business 目录下添加后台管理业务模块，在 api 目录下添加面向客户端的 API 接口，目录结构清晰规范
- 场景 C: 开发者查看项目名称、模块名、配置文件，所有标识统一为 HAB，无混淆

## 背景

本项目从一个具体的银行业务系统（bank）中剥离而来，虽然已经移除了大部分业务逻辑和数据，但仍残留了 GVA 框架的示例代码（example 模块）、不再使用的插件（公告、邮件）、以及业务目录中的零散文件。项目标识（hz-admin-base）也需要统一为新的命名（HAB），使其真正成为一个可复用的基础模板。
