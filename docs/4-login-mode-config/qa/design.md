# qa 技术方案 — 登录模式配置

> 需求: login-mode-config | 角色: qa

## 测试策略

手动功能测试为主，覆盖三种模式的完整登录流程。通过修改 `config.local.yaml` 中的 `login-mode` 值切换模式，每次切换后需重启后端服务。

## 测试范围

### 1. 接口测试: GET /auth/login-mode

| 用例 | 操作 | 预期结果 |
|------|------|----------|
| 返回配置值 | 配置 `login-mode: simple`, 请求 GET /auth/login-mode | 返回 `{"login_mode": "simple"}` |
| 返回配置值 | 配置 `login-mode: captcha` | 返回 `{"login_mode": "captcha"}` |
| 返回配置值 | 配置 `login-mode: strict` | 返回 `{"login_mode": "strict"}` |
| 默认值 | 不配置 login-mode 字段 | 返回 `{"login_mode": "simple"}` |
| 无需认证 | 不带任何 token 请求 | 正常返回 200 |

### 2. simple 模式登录

| 用例 | 操作 | 预期结果 |
|------|------|----------|
| 正常登录 | 输入正确用户名+密码, 点击登录 | 登录成功, 跳转到首页, 获得有效 JWT |
| UI 检查 | 打开登录页 | 无验证码输入框, 无 Passkey 按钮, 无分步流程 |
| 密码错误 | 输入错误密码 | 提示密码错误 |
| 用户不存在 | 输入不存在的用户名 | 提示错误 |
| Token 有效性 | 登录成功后访问需认证接口 | 正常访问 |

### 3. captcha 模式登录

| 用例 | 操作 | 预期结果 |
|------|------|----------|
| 正常登录 | 输入正确用户名+密码+验证码, 点击登录 | 登录成功, 跳转到首页 |
| UI 检查 | 打开登录页 | 显示验证码输入框, 无 Passkey 按钮, 无分步流程 |
| 验证码错误 | 输入错误验证码 | 提示验证码错误, 自动刷新验证码 |
| 不输入验证码 | 留空验证码, 点击登录 | 表单校验提示输入验证码 |
| 验证码刷新 | 点击验证码图片 | 获取新验证码 |

### 4. strict 模式登录

| 用例 | 操作 | 预期结果 |
|------|------|----------|
| 分步流程 | 输入用户名, 点击下一步 | 进入 AUTH 步骤, 显示安全状态 |
| 未绑定用户 | 密码验证通过 | 弹出绑定 TOTP/Passkey 对话框 |
| TOTP 登录 | 已绑定 TOTP 的用户输入验证码 | 登录成功 |
| Passkey 登录 | 已绑定 Passkey 的用户 | Passkey 认证后登录成功 |
| 功能无回归 | 全部 strict 流程 | 与改造前行为完全一致 |

### 5. 跨模式一致性

| 用例 | 操作 | 预期结果 |
|------|------|----------|
| Token 格式 | 三种模式分别登录成功 | simple/captcha 返回标准 JWT; strict 返回 pwdToken (现有行为) |
| 路由加载 | 三种模式登录后 | 动态路由正常加载 |
| 权限控制 | 三种模式登录后访问受限资源 | 权限控制一致 |

## 测试环境配置

```yaml
# 切换模式: 修改 server/config.local.yaml 中 system.login-mode
# 可选值: simple | captcha | strict
# 修改后需重启后端服务
```

## 关键决策

1. 以手动测试为主，因为涉及 UI 交互和浏览器 API (WebAuthn)
2. strict 模式的回归测试是重点，确保改造不影响现有功能
