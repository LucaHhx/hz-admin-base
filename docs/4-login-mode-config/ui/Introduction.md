# 前端实现指南

> 需求: login-mode-config | 由 UI 设计师编写，供前端工程师参考

## 设计理念

本次设计**沿用现有登录页的蓝白配色和左右分栏布局**，三种登录模式共用同一套视觉框架，仅表单内容因模式而异。目标是让用户在不同模式下获得一致的视觉体验，同时保持最小代码改动量。

核心原则:
- **模式透明**: 用户无需知道当前处于哪种模式，UI 自然引导完成登录
- **渐进增强**: simple 最简 → captcha 加验证码 → strict 完整分步流程
- **零破坏**: strict 模式与现有 UI 完全一致，生产环境无感

## 页面布局说明

### 整体结构

```
┌─────────────────────────────────────────────┐
│ [语言切换器]                          (右上角) │
├──────────────────┬──────────────────────────┤
│                  │                          │
│   Logo           │                          │
│   应用名称        │     装饰区域              │
│   副标题          │     (仅桌面端显示)         │
│                  │                          │
│   [表单区域]      │     安全图标               │
│   (根据模式变化)   │     标题文字               │
│                  │     说明文字               │
│                  │                          │
└──────────────────┴──────────────────────────┘
```

### 断点规则

| 断点 | 宽度 | 布局 |
|------|------|------|
| 移动端 | < 768px | 全宽单栏，仅显示表单区域，白色背景 |
| 平板+ | >= 768px (md:) | 左右分栏，左侧50%表单 + 右侧50%装饰 |
| 桌面 | >= 1024px (lg:) | 左侧40%表单 + 右侧60%装饰 |

### 表单卡片

- 最大宽度: `max-w-sm` (384px)
- 水平居中于左侧区域
- 内边距: `px-6 py-12`

## 三种模式的表单差异

### Simple 模式 (`login-mode: simple`)

表单元素:
1. 用户名输入框
2. 密码输入框
3. 登录按钮

不显示: 验证码、Passkey 按钮、分步流程

### Captcha 模式 (`login-mode: captcha`)

表单元素:
1. 用户名输入框
2. 密码输入框
3. 验证码输入框 + 验证码图片（并排）
4. 登录按钮

验证码交互:
- 点击验证码图片刷新
- 验证码图片宽度 `w-32`（128px），高度与输入框等高 `h-11`
- 登录失败且涉及验证码错误时自动刷新验证码
- 验证码图片下方可选显示「点击图片刷新验证码」提示文字

不显示: Passkey 按钮、分步流程

### Strict 模式 (`login-mode: strict`)

保持现有的全部流程，不做任何改动:
- 第一步: 用户名 + 下一步按钮（圆形渐变蓝按钮）
- 查询安全状态后进入第二步
- 第二步: 根据 has_totp / has_passkey 显示不同认证方式
- 未绑定时显示密码+验证码，验证通过后弹出绑定弹窗
- Passkey 按钮始终显示

## 关键交互说明

### 模式获取

前端在页面加载时调用 `/auth/login-mode` 接口获取当前模式，根据返回值决定显示哪套表单。此接口无需 token。

### 输入框

- 使用 Element Plus `el-input` 的 `size="large"` 保持 44px 高度
- 聚焦时显示蓝色边框 + 浅蓝色 ring（Element Plus 默认行为）

### 按钮状态

- 主按钮正常态: `bg-primary-500`，圆角 `rounded-lg`
- 悬停态: `bg-primary-600`
- 加载态: 按钮变淡（`bg-primary-400`），显示旋转图标 + "登录中..."，禁用点击
- 表单输入框在加载时变为灰色背景 `bg-gray-50` 禁用态

### 错误提示

- 使用红色边框高亮错误字段: `border-red-300` + `bg-red-50/50`
- 字段下方显示红色错误文字: `text-xs text-red-500`
- 全局错误使用红色 Toast 横幅显示在表单上方

### 验证码刷新（captcha / strict 模式）

- 点击验证码图片触发刷新
- 登录失败时如果是验证码错误，自动清空验证码输入并刷新图片

## 资源使用指南

**本地资源优先**: 前端实现时必须优先使用 `Resources/` 目录中提供的资源文件，禁止用外部 URL 替代本地应有的资源。

### 可用资源

- `Resources/icons/*.svg` — 纯色 stroke 图标，支持 `currentColor`，可通过 CSS 变色
- `Resources/tokens.css` — CSS 变量，包含所有设计 token（颜色、间距、字号、阴影等）
- `Resources/tailwind.config.js` — Tailwind 扩展配置，可直接合并到项目 `tailwind.config.js`

### 图标列表

| 图标文件 | 用途 |
|----------|------|
| `user.svg` | 用户名输入框前缀图标 |
| `lock.svg` | 密码输入框前缀图标 |
| `shield.svg` | TOTP 验证、安全相关图标 |
| `image-captcha.svg` | 验证码输入框前缀图标 |
| `arrow-right.svg` | Strict 模式下一步按钮 |
| `arrow-left.svg` | Strict 模式返回按钮 |
| `key.svg` | Passkey 按钮图标 |
| `globe.svg` | 语言切换器图标 |
| `refresh.svg` | 刷新操作（备用） |

### 在 Element Plus 中使用 SVG 图标

图标以 SVG 文件提供，前端可根据项目实际情况选择引入方式:
- 直接使用 Element Plus 内置图标（如 `User`, `Lock`, `Key` 等与设计稿视觉一致的）
- 或将 SVG 注册为自定义图标组件

## 注意事项

1. **模式判断逻辑应在一个文件内**: 建议在现有 `login/index.vue` 中通过 `v-if` / `v-show` 切换表单，不需要拆分为多个页面
2. **strict 模式代码不需改动**: 现有的分步逻辑、Passkey、绑定弹窗代码完全保留
3. **simple/captcha 模式不需要查询安全状态**: 直接显示表单，直接调用登录接口
4. **captcha 模式复用现有验证码逻辑**: 现有的 `getCaptcha()` 和验证码相关代码可直接复用
5. **375px 宽度测试**: 确保移动端无水平滚动，表单全宽显示
