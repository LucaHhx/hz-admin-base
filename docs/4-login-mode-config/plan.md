# 登录模式配置

> 通过配置切换登录严格程度，支持不同部署场景（开发/内部/生产）

## 目标

1. 支持三种登录模式：简单模式、验证码模式、强验证模式
2. 通过后端配置 `login-mode` 控制登录行为，前端自动适配
3. 三种模式登录后的 token、路由加载、权限控制均一致

## 范围

**包含:**

- 支持通过配置切换登录模式（simple / captcha / strict）
- 登录页根据当前模式自动适配 UI 和流程
- 提供公开接口供前端查询当前登录模式
- 三种模式登录后的会话管理和权限体系保持一致

**不包含:**

- 登录模式的后台管理界面（当前阶段通过配置文件修改）
- 用户注册流程变更
- TOTP/Passkey 绑定流程变更（仅 strict 模式使用已有流程）

## 核心用户场景

- 场景 A（开发环境）：管理员设置 `login-mode: simple`，用户输入用户名+密码直接登录，无验证码、无分步流程
- 场景 B（内部部署）：管理员设置 `login-mode: captcha`，用户输入用户名+密码+图片验证码登录
- 场景 C（生产环境）：管理员设置 `login-mode: strict`，保持当前分步认证流程（用户名 → 安全状态 → TOTP/Passkey认证 → 未绑定则强制绑定）

## 三种登录模式

| 模式 | 配置值 | 行为 |
|------|--------|------|
| 简单模式 | `simple` | 用户名 + 密码直接登录，无验证码 |
| 验证码模式 | `captcha` | 用户名 + 密码 + 图片验证码登录 |
| 强验证模式 | `strict` | 当前流程：分步认证 + 必须绑定 TOTP/Passkey |

**默认值：`simple`**

## 背景

当前登录流程是固定的"强验证"模式：输入用户名 → 查询安全状态 → 密码/TOTP/Passkey认证 → 未绑定强验证则强制绑定。在开发和内部部署场景中，这种强制绑定流程过于繁琐，需要支持通过配置切换登录严格程度。

## 关键业务决策

1. 登录模式由后端配置集中控制，前端自动适配，不硬编码模式
2. 所有模式登录成功后获得等价的会话凭证，后续权限体系无差异
3. 配置默认值为 `simple`，降低开发和内部部署的使用门槛
4. strict 模式完全保留现有流程，确保生产环境零影响

## 验收清单

- [ ] 设置 `login-mode: simple` → 登录页只显示用户名+密码，登录成功获得有效 JWT
- [ ] 设置 `login-mode: captcha` → 登录页显示用户名+密码+验证码，登录成功获得有效 JWT
- [ ] 设置 `login-mode: strict` → 登录页保持当前分步流程，功能不变
- [ ] 三种模式登录后的 token 格式、路由加载、权限控制均一致
- [ ] `/auth/login-mode` 接口无需 token 即可访问
- [ ] 未配置 login-mode 时默认使用 simple 模式
